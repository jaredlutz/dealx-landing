## Learned User Preferences

- When migrating a site from production, grab ALL images and use every one in the rebuilt UI
- Crawl must capture both `<img>` tags and `background-image` URLs from inline styles
- Content pages must match the original structure (e.g. About: Hero, Mission, Stats, Values, Journey, Testimonials), not generic title+body
- Site is public; only `/admin` requires auth
- Prefer internal links over external diversyfund.com URLs when we have the page; keep top nav minimal (avoid crowding, no Portal login block in footer)
- TopNav **Log In** uses **`getSignInUrl()`** (direct to the investor portal); do not add an on-site **`/portal-login`** landing page unless the user explicitly requests it again
- Mariposa approach for site migration: URL mapping, content migration, image crawl, seed scripts; default imagery work to one new or updated asset and one section per pass unless the user explicitly widens scope (e.g. hero-only means no other sections)
- Brand the markdown insights hub as **Insights & Education** (`/insights-education`, `lib/blog.js`, `content/blog/`); use that label in primary chrome rather than Blog or Journal unless directed otherwise; when Insights is in the header, **`InsightsEducationMegaMenu`** (`components/navigation/InsightsEducationMegaMenu.jsx`) supplies the hover/focus mega panel from **`lib/insights-education-nav.js`**; TopNav/footer may still omit Insights & Education and Premier program links when product calls for minimal chrome (routes remain available by URL)
- Prefer scoped landing-section changes; broad restyles that edit shared UI primitives (Button, Card, SectionTitle, Badge) ripple across admin and other routes and have been reverted—confirm before repeating that pattern; do not reintroduce removed hero experiments (chat UI, scroll-docked sidebar) without an explicit ongoing product request
- Public landing should feel institutional and understated (serious private-markets, high-trust) rather than retail fintech, consumer growth funnels, or flashy marketing-page tropes; when a program or special route should match the home site, use MainSiteChrome and shared tokens rather than a standalone page-only palette; firm-level pages (`/about`, hero positioning) must stay **asset-class agnostic** ("private-market investment platform")—multifamily is one historical chapter, not the identity—and avoid instrument-specific securities language ("promissory notes", "fixed-income structures") in favor of discipline/underwriting framing; brand tone follows the **`Diversyfund Tone & Narrative for Communications.docx`** voice (stoic operator, resolute, fact-stating); when adapting Craig Cecilio's LinkedIn copy, **never reference CXC Experience**; long educational pages (e.g. `/insights-education/{income-vs-growth,ira,retirement-accounts}`, documents, offering circular) should avoid the recurring narrow `max-w-3xl` header + wider `max-w-5xl` card-grid "jiggle"—use either a unified left-anchored width or the editorial two-column layout with sticky TOC (see workspace facts for `EducationalPageToc`)
- Contact and support left columns should follow the same InstitutionalFormPage pattern (eyebrow, primary routing callout, icon highlights) and stay scannable; if CMS pastes extra copy into `contact` body, intro rendering truncates after the intended closing sentence—still clean the block in admin; for dedicated investor contact on program pages, use representative (not advisor) when the user directs
- Home IRA guide teaser (**`components/sections/IraInvestingCta.jsx`**) centers the **geometry-only** DiversyFund mark (**`/images/diversyfund-mark.svg`** in light UI, **`/images/diversyfund-mark-white.svg`** in dark), not the full word lockup—the user adjusts mark scale by request
- `/premier-program` (digital) is invitation-only intake: **noindex** / **robots** disallow; same shell as in-person; server loads **`GET .../premier-booking-url?variant=call`** via **`lib/premier-crm.js`** and passes **`crmBookingCall`**; after form submit, **section 04 — Book a call** (phone only, opens CRM scheduling). **`/premier-program`** → **`POST /api/premier-seat-request`** → df-crm **`/api/public/marketing-site/premier-program`** with **`PREMIER_PROGRAM_WEBHOOK_URL`** + **`PREMIER_PROGRAM_WEBHOOK_SECRET`** (legacy **`PREMIER_SEAT_REQUEST_*`** if unset); **`intakeVariant`** typically **`premier_program`**. **`/premier-program/in-person`**: program detail first, **intake at the bottom** (then scheduling); after submit, two CRM links from **`GET .../premier-booking-url?variant=in_person|call`** via **`lib/premier-crm.js`** (`variant=call` defaults to **`/book/investor-call`** on the CRM host unless **`PREMIER_PROGRAM_BOOKING_CALL_WORKFLOW_ID`** is set on df-crm); tiers and “what you receive” precede intake; same **noindex** / **robots** disallow. df-crm tags **`Marketing_Premier_Program`** / **`Marketing_Premier_InPerson`** (map to workflows in comms settings); optional **`DF_WEBSITE_PREMIER_PROGRAM_WORKFLOW_CAMPAIGN_ID`** on CRM (legacy **`FIXED_NOTE_LP_PREMIER_PROGRAM_WORKFLOW_CAMPAIGN_ID`**); optional **`PREMIER_PROGRAM_BOOKING_WORKFLOW_ID`** / **`PREMIER_PROGRAM_BOOKING_CALL_WORKFLOW_ID`**. Keep tier blocks number-forward and playbook-aligned (Premier tier naming, senior leadership—not CEO-by-name); private-invitation timeline copy in **`lib/premier-invitation-display.js`** uses **"the current offering"** (not "current acquisition pipeline") in both the existing-clients RM-call step and the new-prospective senior-team-reachout step

Middleware coverage: `/api/lead-signup(/*)` is in `shouldRunAuthkit` in `middleware.js`. **Email-verification follow-up step** — when WorkOS's environment-level password policy requires email verification, `authenticateWithPassword` after `createUser` throws `email_verification_required` with a `pending_authentication_token` + `email_verification_id` and WorkOS auto-emails a one-time code. The heuristics (≥10 chars, ≥3 character classes, no common words / repeated runs / sequential keyboard runs) approximate WorkOS's zxcvbn-backed policy so the visitor sees the rule list flip from `×` to `✓` as they type instead of only learning their password is weak after submit; the same `isPasswordStrong` runs server-side in both POST routes. **Insights & Education sign-up gate** — every `/insights-education/*` detail page (except the `/insights-education` index) sits behind a real **AuthKit-backed WorkOS sign-up gate that is fully self-hosted on the LP** — the form lives on the article and there is **no redirect to the WorkOS hosted chooser UI** (`getSignUpUrl` is intentionally not used). The legacy `df-insights-unlocked-v1` localStorage flag remains the no-server first-paint fast-path.
